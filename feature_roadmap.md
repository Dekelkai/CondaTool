# CondaTool 功能规划

## 目标

把 CondaTool 从“环境浏览与基础操作工具”推进为“适合日常使用的 Conda 桌面工作台”，同时保持当前产品的轻量、直观和低门槛特点。

## 当前能力盘点

### 已实现功能

1. 环境管理
   - 创建环境
   - 删除环境
   - 克隆环境
   - 重命名环境

2. 环境导入导出
   - 支持 `yml`
   - 支持 `txt`
   - 支持导出时移除 build 号

3. 包信息查看
   - 查看当前环境已安装包
   - 搜索已安装包
   - 展示版本与渠道

4. 基础桌面体验
   - 深浅色主题与跟随系统
   - 中英文切换
   - 实时日志输出
   - 运行时缺失提示

5. 配置与诊断基础能力
   - 诊断页
   - 当前渠道配置查看
   - 官方默认源 / 清华镜像源模板切换

### 当前边界

当前产品更像“环境级管理 MVP”，适合以下场景：

1. 快速查看本机环境
2. 新建、删除、克隆环境
3. 导入导出环境文件
4. 查看某个环境已安装包

但它还不能完整覆盖日常 Conda 运维场景，例如：

1. 安装、卸载、更新包
2. 配置镜像源
3. 配置代理
4. 处理 pip 混装
5. 管理 Jupyter Kernel
6. 查看当前配置和诊断信息

## 功能合理性判断

### 合理的部分

1. 主界面结构合理
   - 环境、包列表、日志三栏非常符合 Conda 的日常心智模型

2. 现有环境生命周期能力完整度尚可
   - 创建、删除、克隆、重命名、导入导出已经形成一个基础闭环

3. 主题和语言切换合理
   - 作为桌面工具，这类个性化能力应保留，但不需要继续优先扩展

### 不足的部分

1. 包管理链路不闭环
   - 现在能看包，不能真正管理包

2. 网络与源配置缺失
   - 一旦涉及联网安装，用户就会遇到代理、镜像、证书等问题

3. 配置可解释性不足
   - 当前缺少统一诊断页，用户很难理解应用到底使用了哪套配置

4. Python 生态配套能力不足
   - pip、Jupyter、环境变量等都还没有覆盖

## 产品定位建议

建议明确定位为：

> 面向日常开发者、学习者和数据用户的 Conda/Python 环境桌面管理工具。

这个定位意味着优先级应落在：

1. 环境可视化管理
2. 包管理闭环
3. 网络与源配置
4. Python 生态配套

而不是先做太重的高级分析能力。

## 规划原则

1. 先补闭环，再扩能力
2. 先解决高频真实场景，再做锦上添花
3. 后端统一 `conda` / `micromamba` 差异，前端只消费统一数据结构
4. 所有全局配置修改都要明确提示作用范围
5. 所有破坏性操作都要有确认与可追踪日志

## 推荐路线

### 阶段 A：诊断与设置基础设施

目标：让用户知道应用当前到底在使用什么配置。

建议新增页面：`设置 / 诊断`

建议能力：

1. 诊断总览
   - 当前包管理器类型
   - 当前 root/base 路径
   - 当前活动环境
   - Conda / micromamba / Python 版本
   - envs 目录
   - config 文件来源
   - 是否启用代理
   - channels 摘要

2. 配置来源展示
   - `.condarc`
   - `.mambarc`
   - 环境变量覆盖

3. 一键导出诊断信息

优先级：P0

原因：

1. 这是源配置、代理配置、包安装等功能的基础设施
2. 能显著减少“为什么和命令行不一样”的问题

### 阶段 B：源配置

目标：解决最常见的 Conda 实际使用痛点。

建议能力：

1. 查看当前 channels
2. 新增 channel
3. 删除 channel
4. 调整 channel 顺序
5. 设置 channel priority
6. 一键切换镜像模板
   - defaults
   - conda-forge
   - 清华
   - 中科大
7. 恢复默认源
8. 测试源可用性

建议 UI：

1. 设置页下新增“源配置”标签
2. 列表展示当前顺序
3. 模板切换与手动编辑分开

优先级：P0

原因：

1. 直接影响安装、更新、创建环境的成功率与速度
2. 是真实用户使用频率很高的能力

### 阶段 C：代理配置

目标：补齐企业/校园/国内网络环境的必需能力。

建议能力：

1. 查看当前 HTTP/HTTPS 代理
2. 设置 HTTP/HTTPS 代理
3. 清除代理
4. 设置 `ssl_verify`
5. 网络连通性测试
6. 展示配置生效范围

注意点：

1. 代理配置通常是用户级而非环境级
2. 如有用户名密码，要明确安全风险

优先级：P0

原因：

1. 联网操作离不开它
2. 没有代理入口，很多用户只能手改配置文件

### 阶段 D：包管理闭环

目标：把“只读包列表”升级为真正的包管理面板。

建议能力：

1. 安装包
2. 卸载包
3. 升级包
4. 指定版本安装
5. 远程包搜索
6. 展示候选版本与来源 channel
7. 明确区分 conda / pip 包来源

建议 UI：

1. 包列表顶部新增操作栏
2. 已安装包支持卸载/升级入口
3. 安装弹窗支持输入包名与版本选择

优先级：P0

原因：

1. 当前产品最大的不完整点就是“能看不能管”
2. 这是用户最直接能感知的功能缺口

### 阶段 E：pip 与 Jupyter 生态支持

目标：覆盖真实 Python 环境工作流。

建议能力：

1. pip 包安装/卸载
2. requirements 导出
3. pip 包来源标识
4. Jupyter Kernel 注册
5. Jupyter Kernel 删除
6. Kernel 与环境映射展示

优先级：P1

原因：

1. 很多用户并不是纯 conda 工作流
2. Notebook 用户对 kernel 管理有明显需求

### 阶段 F：环境变量与高级工具

目标：补足长期维护能力。

建议能力：

1. 环境变量管理
2. 包缓存清理
3. 操作历史
4. 常见诊断导出
5. dry-run 预览
6. 依赖冲突辅助信息

优先级：P1 / P2

## 优先级汇总

1. 历史记录

## 后端接口扩展建议

基于当前 `backend/main.py` 的结构，建议继续沿用“子命令 + JSON 输出”的模式。

建议新增命令组：

1. `config-get`
2. `config-set-proxy`
3. `config-clear-proxy`
4. `channel-list`
5. `channel-add`
6. `channel-remove`
7. `channel-move`
8. `channel-reset`
9. `pkg-install`
10. `pkg-remove`
11. `pkg-upgrade`
12. `pkg-search`
13. `pip-install`
14. `pip-remove`
15. `kernel-list`
16. `kernel-add`
17. `kernel-remove`
18. `env-vars-list`
19. `env-vars-set`
20. `env-vars-remove`

## 关键风险

1. `conda` 与 `micromamba` 配置语义不同
2. 源与代理配置是全局行为，不是环境行为
3. pip 与 conda 混装需要清晰标识来源
4. 涉及代理账号密码时要处理安全提示
5. 包安装和搜索很容易带来长任务，需要继续利用现有日志体系

## 里程碑建议

### M1：配置可见

完成后，用户至少能回答：

1. 我现在用的是 conda 还是 micromamba？
2. 我的 root/base 在哪里？
3. 我的 channels 和代理是什么？

### M2：网络可控

完成后，用户至少能：

1. 切换镜像
2. 设置代理
3. 测试是否能联网安装

### M3：包管理闭环

完成后，用户至少能：

1. 搜索包
2. 安装包
3. 卸载包
4. 更新包

### M4：Python 生态补齐

完成后，用户至少能：

1. 管理 pip 包
2. 管理 Jupyter Kernel
3. 配置环境变量

## 一句话建议

现阶段最值得先做的不是继续打磨首页，而是按下面顺序推进：

1. 诊断页
2. 源配置
3. 代理配置
4. 包管理闭环

这样才能把 CondaTool 从“能看环境”推进到“能日常使用”。
