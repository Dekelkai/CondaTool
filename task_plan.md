## 关键问题
1. `runCommand` 只依赖 React state 做互斥，无法保证串行执行。
2. 监听器注册与 `probe` 启动存在竞态，可能丢失事件。
3. `pkg-list` 响应没有和当前选中环境绑定，可能显示错误包列表。
4. 高频日志流会导致前端频繁 `setLogs` 重渲染，长命令时可能明显卡顿。
5. 包列表搜索对大环境包数据集会在每次按键同步过滤，响应不够平滑。
6. `env-list` 当前会对多个环境执行 `python --version`，超时后叠加到整体启动耗时。
7. `pkg-list` 当前依赖 `micromamba list --json`，在 base 环境上耗时超过两分钟。
8. `micromamba info/env list` 的语义与 `conda info/env list` 不同，直接透传会把 `mamba` 根前缀误显示为环境。
9. 当前项目尚未覆盖包安装/卸载、源配置、代理配置、诊断页等高频实际能力。
10. 诊断信息缺失会放大配置差异带来的理解成本。
11. 源配置既要兼容 `conda`，也要兼容当前运行时使用的 `micromamba`。
12. 预设源模板需要与真实 `.condarc` 写入结果保持一致，不能依赖包管理器的隐式默认扩展。
13. 另一台机器上的 runtime missing 高概率与未签名的 `backend.exe` / `micromamba.exe` 被拦截有关。

## 已做决策
| 决策 | 理由 |
|------|------|
| 先修前端命令调度与状态同步 | 这是当前最直接、最影响用户操作正确性的故障链 |
| 暂不扩展到跨平台 runtime 架构重构 | 范围过大，超出本轮“最小必要修复” |
| 优先做最小代码改动，不引入新依赖 | 降低回归风险，便于快速验证 |
| 用串行命令队列代替 `running` state 互斥 | `running` 只能驱动 UI，不能保证命令真正串行结束 |
| 把环境选中同步收口到 `env-list` 刷新结果 | 避免删除/重命名成功后前端保留过期路径 |
| 本轮优先优化前端日志渲染与包搜索 | 这是桌面端最容易感知的性能瓶颈，改动面也最小 |
| 把 `env-list` Python 版本探测限制为固定并发 | 避免环境较多时为每个环境创建线程 |
| 优先用本地 `conda-meta` 替代慢速外部命令 | 这是当前运行慢的主要根因，收益远高于继续调前端 |
| 在后端统一归一化包管理器返回结构 | 避免前端承担 `conda`/`micromamba` 差异处理 |
| 用独立路线文档承接产品能力扩展 | 便于后续按阶段推进，而不是把规划散落在对话里 |
| 诊断页先做成弹层而非完整新页面 | 以最小改动尽快落地第一版能力 |
| 源配置优先围绕 `channels/default_channels/custom_channels/channel_priority` 落地 | 这是最核心、最常用、最容易验证的一组能力 |
| 源配置第一版使用预设模板切换，不先做自定义增删改排序 | 这样能保证可用性和发版稳定性 |
| 先接入证书无关的签名脚本 | 在未提供证书前先把发布流程准备好，避免阻塞后续签名 |
